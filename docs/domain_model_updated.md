## 1. Introduction
This document outlines the Domain Model for the Cithara platform. It identifies the core business entities, their attributes, and relationships based on the approved Software Requirements Specification (SRS). The model focuses on the problem domain (music generation and collection) rather than software implementation details.

## 2. Identified Domain Concepts

| Domain Entity | Description | SRS Trace |
|---|---|---|
| User | The base entity represents identity and authentication data common to all roles. | FR-01, US-01 |
| Creator | A specialized User who can generate music and owns a Library. | US-02, Section 2.3 |
| Listener | A specialized User who interacts with the system primarily to listen to shared content. | US-14, Section 2.3 |
| Library | A logical collection of songs owned specifically by a Creator. | US-07, FR-08 |
| Song | The primary artifact generated by the system, containing metadata and audio links. | US-02, FR-08 |
| **GenerationJob** | **[NEW] Tracks the status and results of an asynchronous song-generation task.** | **FR-05, FR-06, UC-02** |
| Suno API | (External System) The third-party AI service used to generate the audio files. | Section 2.1 |
| Google OAuth | (External System) The third-party provider used for user authentication. | FR-01 |

## 3. Domain Model Specification

### 3.1 Domain Classes & Attributes

#### Class: User (Abstract Base Class)
- **Responsibilities**: Represents the core identity of any person accessing the platform.
- **Attributes**:
  - `email`: String
  - `display_name`: String
  - `date_joined`: DateTime
  - `last_login`: DateTime

#### Class: Creator (Inherits from User)
- **Responsibilities**: A specialized user who generates music. This entity has permissions to create new content and manage a personal library.
- **Attributes**:
  - Inherits all attributes from User
  - `generation_quota`: Integer (Daily limit tracking)

#### Class: Listener (Inherits from User)
- **Responsibilities**: A specialized user who consumes content. They interact with the system primarily by listening to shared links.
- **Attributes**:
  - Inherits all attributes from User
  - `recently_played`: List\<Song\> (History tracking)

#### Class: Library
- **Responsibilities**: Represents the personal collection of a specific User. It acts as the container for all Songs owned by that Creator.
- **Attributes**:
  - `total_songs_count`: Integer (derived)
  - `date_created`: DateTime
- **Relationships**:
  - One Library belongs to exactly one Creator (1:1)

#### Class: Song
- **Responsibilities**: The core artifact generated by the system. It stores the descriptive metadata that defines the track.
- **Attributes**:
  - `title`: String
  - `duration`: Integer (in seconds)
  - `genre`: Enum (Rock, Pop, Jazz, Hiphop, Country)
  - `mood`: Enum (Happy, Sad, Energetic, Calm)
  - `occasion`: Enum (Celebration, Funeral, Relaxation, Workout)
  - `audio_url`: URL (link to the generated audio file)
  - `prompt`: Text (original prompt used for generation — stored for regeneration, UC-08)
  - `style`: String (original style used for generation — stored for regeneration, UC-08)
  - `rating`: Integer 1–5 (user rating, nullable — FR-16)
  - `date_generated`: DateTime
  - `is_deleted`: Boolean (soft delete flag — hidden from Library without DB removal)
- **Relationships**:
  - One Song belongs to exactly one Library (N:1)

#### Class: GenerationJob *(NEW)*
- **Responsibilities**: Tracks a song-generation task submitted through any strategy. Links to the Song record that was (or will be) produced. Enables async polling of the external AI API.
- **Attributes**:
  - `task_id`: String (unique identifier returned by the generation strategy)
  - `status`: Enum (PENDING, TEXT_SUCCESS, FIRST_SUCCESS, SUCCESS, FAILED)
  - `strategy_used`: Enum (Mock, Suno)
  - `audio_url`: URL (audio file URL from the AI service)
  - `stream_audio_url`: URL (streaming audio URL)
  - `image_url`: URL (cover art URL from the AI service)
  - `error_message`: String (error details if generation failed)
  - `created_at`: DateTime
  - `updated_at`: DateTime
- **Relationships**:
  - One GenerationJob belongs to exactly one Song (N:1) — multiple jobs can be created for retries
- **SRS Trace**: FR-05 (progress indicator), FR-06 (15-minute timeout enforcement)

#### External Entity: Google OAuth 2.0
- **Responsibilities**: Verifies user identity and handles secure login, returning the verified email and name to the system.

#### External Entity: Suno API
- **Responsibilities**: The external AI engine that receives the conceptual parameters (Genre, Mood, Occasion) and synthesizes the final audio track.

## 4. Key Assumptions and Justifications
- **Single vs. Multiple Attributes**: A generated Song is categorized by exactly one primary Genre, one primary Mood, and one specific Occasion.
- **Library Cardinality**: A Creator has exactly one Library that holds all their generated tracks.
- **Soft Deletion**: When a user "deletes" a song, the `is_deleted` flag is set to True, hiding it from the Library view without removing the database record.
- **GenerationJob Lifecycle**: A GenerationJob is created at generation time and updated during async polling. Once status reaches SUCCESS or FAILED, the job is finalized.
- **External Entity Boundaries**: Google OAuth and Suno API are treated as external boundaries. We assume they consistently provide the conceptual outputs we need.
